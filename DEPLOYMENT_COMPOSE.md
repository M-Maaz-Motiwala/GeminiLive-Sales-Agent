# Docker Compose Deployment Flow

Use `start.sh` instead of calling `docker compose` directly. The first argument selects the environment and `start.sh` automatically adds the matching compose override file.

It also selects the matching env file:

| Environment | Env file         |
| ----------- | ---------------- |
| local       | `.env`         |
| prod        | `.env.prod`    |
| staging     | `.env.staging` |

## Start Commands

```bash
./start.sh local up -d --build
./start.sh staging up -d --build
./start.sh prod up -d --build
```

Equivalent environment-variable form:

```bash
DEPLOY_ENV=local ./start.sh up -d --build
DEPLOY_ENV=staging ./start.sh up -d --build
DEPLOY_ENV=prod ./start.sh up -d --build
```

Common operations:

```bash
./start.sh staging ps
./start.sh staging logs -f platform
./start.sh staging logs -f bridge
./start.sh staging down
./start.sh staging up -d --force-recreate asterisk
./scripts/check.sh staging
```

Make targets also accept `ENV`:

```bash
make up ENV=staging
make check ENV=prod
make logs ENV=staging
make down ENV=staging
```

## Compose Files

- `docker-compose.yml`: shared service definitions.
- `docker-compose.local.yml`: local container names and ports.
- `docker-compose.staging.yml`: staging container names and ports.
- `docker-compose.prod.yml`: production container names and ports.

`start.sh staging ...` runs:

```bash
docker compose -f docker-compose.yml -f docker-compose.staging.yml -p aura_staging ...
```

`start.sh prod ...` runs:

```bash
docker compose -f docker-compose.yml -f docker-compose.prod.yml -p aura_prod ...
```

The `-p` project name and environment-specific `container_name` values prevent collisions when prod and staging run on the same Docker host.

## Ports

| Environment | Platform API | Frontend | Asterisk ARI | SIP UDP | Asterisk RTP UDP |
| ----------- | -----------: | -------: | -----------: | ------: | ---------------- |
| local       |         8000 |     8080 |         8088 |    5060 | 10000-10050      |
| prod        |         8000 |     8080 |         8088 |    5060 | 10000-10050      |
| staging     |         8001 |     8081 |         8089 |    5061 | 10060-10110      |

Bridge RTP ports are separate internal bridge media ports and currently remain controlled by:

```bash
RTP_PORT_BASE=40000
RTP_PORT_COUNT=50
```

Do not confuse those with the public Asterisk RTP range above.

## Firewall Rules

Production:

```bash
sudo ufw allow 8000/tcp
sudo ufw allow 8080/tcp
sudo ufw allow 5060/udp
sudo ufw allow 10000:10050/udp
```

Staging:

```bash
sudo ufw allow 8001/tcp
sudo ufw allow 8081/tcp
sudo ufw allow 5061/udp
sudo ufw allow 10060:10110/udp
```

ARI should normally not be public. If DevOps needs temporary ARI access, restrict it to a trusted IP or VPN:

```bash
sudo ufw allow from <trusted-ip> to any port 8088 proto tcp
sudo ufw allow from <trusted-ip> to any port 8089 proto tcp
```

## Environment Variables

Each environment uses its own env file plus an auto-generated host env file. `start.sh` selects both based on the `local`/`staging`/`prod` argument:

| Environment | App env file     | Auto-generated host env file |
| ----------- | ---------------- | ---------------------------- |
| local       | `.env`         | `.host.local.env`          |
| prod        | `.env.prod`    | `.host.prod.env`           |
| staging     | `.env.staging` | `.host.staging.env`        |

> [!NOTE]
> All `.host.*.env` files are auto-generated by `start.sh` — **do not edit them manually**.
> They contain `EXTERNAL_IP`, `SIP_PORT`, and NAT settings resolved at startup time.
> Both `.host.*.env` and `asterisk/generated_*/` are git-ignored to prevent cross-environment contamination.

### Generated Dialplan Directories

Each environment writes CRM-registered DIDs to its own isolated directory on the host:

| Environment | Host path                       | Container path               |
| ----------- | ------------------------------- | ---------------------------- |
| local       | `asterisk/generated_local/`   | `/etc/asterisk/generated/` |
| prod        | `asterisk/generated_prod/`    | `/etc/asterisk/generated/` |
| staging     | `asterisk/generated_staging/` | `/etc/asterisk/generated/` |

`start.sh` creates the host directory automatically before starting Docker.

### APP_ENV and Container Names

Set `APP_ENV` in each environment's env file so the platform and diagnostics know which Docker containers to target:

```bash
# .env.prod
APP_ENV=prod
ASTERISK_CONTAINER_NAME=aura_prod_asterisk
BRIDGE_CONTAINER_NAME=aura_prod_bridge

# .env.staging
APP_ENV=staging
ASTERISK_CONTAINER_NAME=aura_staging_asterisk
BRIDGE_CONTAINER_NAME=aura_staging_bridge
```

These values drive:

- Admin UI **Telephony Diagnostics** (docker socket checks, container status, log viewer).
- Backend **dialplan reload** (`asterisk -rx "dialplan reload"` after CRM org DID registration).

`start.sh staging ...` exports `APP_ENV_FILE=.env.staging` and `HOST_ENV_FILE=.host.staging.env`.

You normally do not need to set these manually. Use only for CI/CD or a custom env path:

```bash
APP_ENV_FILE=/opt/aura/prod/.env ./start.sh prod up -d --build
APP_ENV_FILE=/opt/aura/staging/.env ./start.sh staging up -d --build
```

For SIP ports:

```bash
SIP_PORT=5060
PROD_SIP_PORT=5060
STAGING_SIP_PORT=5061
```

For Asterisk RTP:

```bash
ASTERISK_RTP_START=10000
ASTERISK_RTP_END=10050
```

Staging overrides Asterisk RTP in `docker-compose.staging.yml`:

```yaml
ASTERISK_RTP_START: 10060
ASTERISK_RTP_END: 10110
```

Production contains the live DIDWW trunk values in `.env.prod`.

Staging intentionally leaves trunk credentials commented/empty in `.env.staging` until a staging DID/trunk exists. Keep staging on `OUTBOUND_MODE=lab` unless a staging route is ready.

Recommended production/staging practice:

```text
/opt/aura/prod/.env.prod
/opt/aura/staging/.env.staging
```

The checked-out local files `.env.prod` and `.env.staging` are ignored by git because they can contain secrets. Commit only templates/docs, not real env files.

## DID And Trunk Routing

Production and staging should not share the same live DID route.

Recommended:

- Production DID routes to `SERVER_IP:5060`.
- Staging test DID routes to `SERVER_IP:5061`.
- Production trunk credentials stay in prod env.
- Staging trunk credentials/DID stay in staging env, if your provider requires separate credentials.

If the SIP provider supports multiple DID routes on the same trunk, you may not need a completely separate trunk, but you still need a separate staging DID or a clear routing rule that sends test calls to `:5061`.

If the provider/company Asterisk cannot route by port or test DID, create a separate staging trunk/DID. That is the safest option and avoids staging calls landing in production.

## Quick DevOps Checklist

- Start prod with `./start.sh prod up -d --build`.
- Start staging with `./start.sh staging up -d --build`.
- Open prod ports: `8000/tcp`, `8080/tcp`, `5060/udp`, `10000:10050/udp`.
- Open staging ports: `8001/tcp`, `8081/tcp`, `5061/udp`, `10060:10110/udp`.
- Keep ARI ports `8088/tcp` and `8089/tcp` private or IP-restricted.
- Route prod DID/trunk to `SERVER_IP:5060`.
- Route staging DID/trunk to `SERVER_IP:5061`.
- Run `./scripts/check.sh prod` and `./scripts/check.sh staging` after deployment.
